# Internationalisation (i18n) conventions

## Stack

- **i18next** + **react-i18next** + **i18next-http-backend** for all translations
- **Never hardcode user-visible strings** — every string that reaches the UI must come from a translation file

## Supported locales

| Code | Language | Default |
|---|---|---|
| `en` | English | ✅ Yes |
| `es` | Spanish | |
| `pt-BR` | Portuguese (Brazil) | |

**Default language is English (`en`).** `fallbackLng` is also `en`.

## File structure

```
public/
└── locales/
    ├── en/
    │   ├── common.json
    │   └── <feature>.json
    ├── es/
    │   ├── common.json
    │   └── <feature>.json
    └── pt-BR/
        ├── common.json
        └── <feature>.json

src/
└── i18n.ts    # i18next setup — import this once in main.tsx
```

Translation files live in `public/locales/` and are served as static assets — **no recompile needed** when changing a translation.

## Rule: always add all three locales

**Every time you add a new key, you must add it in all three locale files simultaneously** — `en`, `es`, and `pt-BR`. Never add a key to one locale without adding it to the others. Omitting a locale will cause the fallback (`en`) to show for that key, which is acceptable temporarily, but the key must still exist in all files.

## Namespace convention

| Namespace | File | When to use |
|---|---|---|
| `common` | `common.json` | Brand name, generic errors, shared UI labels |
| `attendee` | `attendee.json` | Attendee-facing screens |
| `admin` | `admin.json` | Admin-facing screens |
| `<feature>` | `<feature>.json` | One namespace per feature domain |

When creating a new namespace, register it in `src/i18n.ts` under the `ns` array.

## Translation key structure

Use nested objects grouped by screen/component, then by element:

```json
{
  "registration": {
    "title": "...",
    "fields": {
      "email": {
        "label": "...",
        "placeholder": "...",
        "errors": {
          "required": "...",
          "invalid": "..."
        }
      }
    }
  }
}
```

Keys must be in English, values in the target language.

## Using translations in components

```tsx
import { useTranslation } from 'react-i18next'

const MyComponent = () => {
  const { t } = useTranslation(['attendee', 'common'])

  return <h1>{t('attendee:registration.title')}</h1>
}
```

- Always declare the namespaces you use in the `useTranslation` call
- Reference other namespaces with the `namespace:key` prefix syntax
- The first namespace in the array is the default (no prefix needed for it)

## Using translations in Zod schemas

Because translations are loaded async (HTTP backend), **schemas must be functions** so that `i18n.t()` is called at validation time, not at module evaluation time:

```ts
import { z } from 'zod'
import i18n from '@/i18n'

export const mySchema = () =>
  z.object({
    name: z.string().min(1, i18n.t('attendee:registration.fields.name.errors.required')),
  })

export type MyValues = z.infer<ReturnType<typeof mySchema>>
```

Call it when setting up the form resolver:

```ts
resolver: zodResolver(mySchema())
```

## Adding a new locale

1. Create `public/locales/<code>/common.json` and `<feature>.json` files
2. Add the locale code to `SUPPORTED_LOCALES` in `src/i18n.ts`

## Setup (already done — do not repeat)

`src/i18n.ts` is imported at the top of `src/main.tsx`. The app is wrapped in `<Suspense>` to handle the async loading of translation files.
