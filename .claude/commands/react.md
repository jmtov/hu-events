# React conventions

## Component structure

- Use functional components ‚Äî **arrow function** style consistently:

```tsx
// YES
const MyComponent = ({ title }: MyComponentProps) => { ... }
export default MyComponent
```

- Keep components small and focused ‚Äî split at ~150 lines
- No logic dangling in components ‚Äî move to utils or custom hooks
- Keep render logic light ‚Äî move complex maps/conditions to utils
- Co-locate related components in a feature folder: `src/features/events/`, `src/features/participants/`, etc.
- Shared/generic UI components live in `src/components/`

## Styling

- Use **shadcn/ui** components as the base for all UI ‚Äî use them in their purest form with minimal customization
- Use **Tailwind CSS** for all styling ‚Äî never use inline `style` props
- If a component needs layout or spacing adjustments, use Tailwind utility classes
- **Never use emojis** in UI text, labels, buttons, or comments unless the user explicitly requests it

## Typography

Use this scale consistently across the app. Never invent ad-hoc sizes.

| Role | Element | Tailwind classes |
|---|---|---|
| Page title | `h1` | `text-2xl font-bold` |
| Section title | `h2` | `text-xl font-semibold` |
| Card / panel title | `h3` | `text-base font-semibold` |
| Body | `p` | `text-sm` |
| Secondary / muted | `p` | `text-sm text-muted-foreground` |
| Caption / hint | `p` | `text-xs text-muted-foreground` |
| Error message | `p` | `text-xs text-destructive` |
| Label (form field) | shadcn `Label` | default ‚Äî do not override size |

## Props

- Always define props with a named `type`: `type EventCardProps = { ... }`
- Never use `React.FC` ‚Äî type the arrow function directly
- Destructure props at the function signature level

## Event handler naming

- Props that accept a handler ‚Üí **`on` prefix**: `onChange`, `onSubmit`, `onClose`
- Internal functions that handle an event ‚Üí **`handle` prefix**: `handleChange`, `handleSubmit`, `handleClose`

## State

- **TanStack Query** for server/async state (data fetched from an API)
- **`useState`** for local UI state (open/closed, selected tab, etc.)
- Never use `useState` for form fields ‚Äî that's TanStack Form's job
- Keep state close to where it's used ‚Äî avoid prop drilling

## Hooks

- Custom hooks live in `src/hooks/` and start with `use`
- A hook that fetches data must use TanStack Query internally
- A hook must do one thing ‚Äî don't combine unrelated logic
- Follow Rules of Hooks ‚Äî use appropriate dependency arrays and implement cleanup when needed

## Performance

- Avoid `useCallback` and `useMemo` unless there is a clear, measurable need ‚Äî do not wrap every handler preemptively
- Use `React.memo` for expensive components
- Always provide a stable, meaningful `key` prop when rendering lists ‚Äî never use array index unless the list is static and never reordered

## Rendering

- Handle `isLoading` and `isError` states explicitly in every component that fetches data
- Use `condition && <Component />` for single-condition rendering ‚Äî avoid `condition ? <Component /> : null`
- Never render user-provided strings as HTML (`dangerouslySetInnerHTML`) unless explicitly required and sanitized

## Accessibility

- Icon-only interactive elements **must** have an `aria-label`
- Use semantic HTML elements ‚Äî do not use `div` with `onClick` as a button substitute
- Use `aria-labelledby` / `aria-describedby` to associate labels and descriptions when the label is visible elsewhere

## Imports

- Use absolute imports (`@/...`) ‚Äî never use `../../../` relative paths more than one level deep
- Import order: external libraries ‚Üí internal modules ‚Üí types

## Icons

Use **`@tabler/icons-react`** for all icons ‚Äî never use emojis as icons, never import from other icon libraries.

```tsx
import { IconCalendar, IconUser } from '@tabler/icons-react'

<IconCalendar size={16} />
```

- Default `size` is 24 ‚Äî use `size={16}` for inline/small icons, `size={20}` for UI controls
- Use `stroke={1.5}` for a lighter look when needed (Tabler default is `2`)
- Icon-only elements must always have an `aria-label`

## shadcn/ui components (`src/components/ui/`)

Files in `src/components/ui/` are **auto-generated by the shadcn CLI** and must not be manually edited. They intentionally break several project conventions (function declarations, named exports, multiple components per file) ‚Äî this is expected and correct.

- Never refactor them to arrow functions or default exports
- When a component needs customisation, wrap it in `src/components/` rather than modifying the `ui/` file directly
- To add a new component: `npx shadcn add <component>`

## What to avoid

- Business logic in the component file ‚Äî use hooks or utils
- Wrapper components only for styling ‚Äî apply classes directly on the component
